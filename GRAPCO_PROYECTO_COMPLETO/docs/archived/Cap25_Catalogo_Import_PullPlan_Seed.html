Reverse scheduling colaborativo. El equipo define hitos finales y "tira" hacia atrás las tareas necesarias. Vista Kanban por hitos con tareas pull, fechas calculadas automáticamente.

25.2 Catálogo Empresarial de APUs (scope híbrido)

Antes del Bloque 25, todos los APUs eran globales (sin contexto). Ahora la arquitectura distingue entre:

ScopeCaracterísticasEjemplo
🌎 empresaReutilizables en TODOS los proyectos. Mantenidos por la empresa."Concreto f'c=210" usado en 5 obras
🏗️ proyectoEspecíficos del proyecto. No interfieren con catálogo empresarial."Vaciado especial PTARI" customizado

25.2.1 Operaciones implementadas

  • Filtro visual por scope (Todos · 🌎 Empresa · 🏗️ Proyecto)
  • 📋 Copiar al proyecto: clona un APU empresarial al proyecto activo, con scope=proyecto. Modificable sin afectar el original. Conserva referencia copiadoDeApuId.
  • 🌎 Promover a empresa: un APU específico se vuelve empresarial (solo admin). Útil cuando un APU custom resulta tan bueno que vale para todos los proyectos.
  • Conteo doble en header: cuántos empresariales y cuántos en el proyecto activo.
✅ Beneficio operacional: El catálogo empresarial crece con la experiencia. Cada proyecto puede contribuir APUs probados al patrimonio empresarial.

25.3.3 Flujo del importador

  1. Usuario arrastra archivo .csv (o hace click) → se lee y parsea
  2. Sistema detecta columnas → muestra qué se mapeó automáticamente
  3. Vista previa de las primeras 15 actividades con totales (presupuesto, partidas hoja)
  4. Usuario selecciona frente destino
  5. Click "Importar" → carga masiva en lotes de 400 con writeBatch

25.4 Pull Planning (LCI fase 2)

25.4.1 La metodología

El Pull Planning es la fase 2 del Last Planner System (Glenn Ballard, 1992). La fase 1 es el Master Schedule (Plan Maestro). La fase 2 es donde el equipo de obra define hitos finales y "jala" hacia atrás las tareas necesarias para cumplirlos. Es colaborativo y compromisivo: cada subcontratista, cuadrilla, oficina técnica se compromete con sus tareas.

📚 Origen: Lean Construction Institute (USA). Implementado por DPR, Turner, Skanska. La sesión típica dura 2-4 horas con todos los actores en una sala (físicamente o virtualmente). Se usan post-its de colores en una pared (digital aquí).

25.4.2 Implementación visual

  • Vista Kanban horizontal: cada hito es una columna ordenada por fecha
  • 3 tipos de hito: 📜 Contractual (no negociable), 🏗️ Fase, 🎯 Intermedio
  • Tareas pull dentro de cada hito, con diasAntesDelHito y duracionDias
  • Fecha calculada: fechaInicio = fechaHito - diasAntesDelHito
  • Prerequisitos: campo de texto donde el equipo escribe qué debe estar listo antes (constraints)
  • Progreso: barra con tareas completadas / totales por hito
  • Indicador "VENCIDO" en rojo si la fecha del hito ya pasó y no está cumplido

25.4.3 Modelo de datos

// PullPlanningHitos/{hitoId}
{
  nombre, descripcion, tipo: 'contractual|fase|intermedio',
  fechaHito: timestamp,
  proyectoId, frenteId, responsable,
  cumplido: boolean, cumplidoEn: timestamp
}

// PullPlanningTareas/{tareaId}
{
  nombre, descripcion,
  hitoId, proyectoId,
  diasAntesDelHito: number,    // reverse scheduling
  duracionDias: number,
  responsableEmpresa, cuadrilla,
  prerequisitos,                // constraints
  color, completada: boolean
}

25.5.1 Cómo funciona

Cada documento creado lleva un campo _seedTag = 'SEED_PTARI_DEMO_2026'. La función limpiarSeed() hace una query where('_seedTag', '==', SEED_TAG) y elimina todo. Esto permite cargar y limpiar el demo sin afectar los datos reales.

💡 Workflow recomendado para tu sustentación:
  1. Inmediatamente antes de la sustentación: Admin → Datos Demo → Cargar PTARI
  2. Sustentas con datos completos y realistas
  3. Después de la sustentación: Admin → Datos Demo → Limpiar (opcional)
