⚠️ Diagnóstico: el riesgo real

Hasta el Bloque 11, las reglas de Firestore y Storage tenían el setting más peligroso posible: allow read, write: if true. Esto significa que CUALQUIERA con la URL del proyecto Firebase podía:

📖

Leer toda la base de datos

Cualquier persona con un cliente HTTP (curl, Postman) y la URL del proyecto Firebase podía descargar todos los registros de obra, costos, personal, modelos BIM y datos confidenciales del proyecto PTARI.

✓ Resuelto: solo usuarios autenticados leen
✏️

Modificar/borrar cualquier dato

Sin reglas, un atacante podía borrar todas las cuadrillas, alterar CPI, cambiar tarifas, manipular registros para reportes falsos. Riesgo CRÍTICO para integridad de la tesis y operación de obra.

✓ Resuelto: escritura requiere rol válido
💸

Inflar la factura de Firebase

Un atacante podía hacer un script que escriba millones de documentos basura en Firestore en minutos, generando costos por GB y por operación. Plan Blaze = cuenta abierta sin protecciones.

✓ Resuelto: validación de tamaños + rate limit
🖼️

Subir archivos arbitrarios

Storage abierto = cualquiera puede subir archivos pesados, malware o contenido ilegal usando tu bucket. Tu cuenta queda asociada a ese contenido.

✓ Resuelto: validación MIME + tamaño máx
🤖

Fuerza bruta en login

Sin rate limiting, un script podía probar millones de contraseñas contra cualquier email. Firebase tiene cierta protección, pero no es perfecta.

✓ Resuelto: 5 intentos / 5 min cooldown
🕵️

Sin trazabilidad

Si algo se borra o altera, no había forma de saber quién lo hizo o cuándo. Imposibilitaba auditorías post-incidente.

✓ Resuelto: log Auditoria_Seguridad

1 Reglas Firestore endurecidas

Reemplazo total del archivo firestore.rules. Pasa de 4 líneas abiertas a 150+ líneas con validación por colección.

Antes vs Después

❌ ANTES
Reglas abiertas (peligroso)
match /{document=**} {
  allow read, write: if true;
}
✅ AHORA
Reglas con roles + validación
match /Registros_Campo/{docId} {
  allow read: if isAuthorized();
  allow create: if isCapataz() &&
    request.resource.data.uidCapataz == uid() &&
    numInRange(metrado, 0, 100000);
  allow delete: if isAdmin();
}

3 Sistema de roles formal

Cada usuario tiene un documento en /Usuarios/{uid} con su rol y campo activo. Las reglas leen este documento para autorizar.

Schema del documento de usuario

{
  email: "juan@grapco.pe",
  nombre: "Juan Pérez",
  rol: "capataz",         // capataz | ingeniero | admin
  activo: true,             // false = bloquear acceso
  creadoEn: serverTimestamp(),
  ultimoLogin: serverTimestamp()
}

3 roles definidos

👷

capataz

Reporta producción de campo. Crea registros, sube fotos, edita SUS borradores. NO puede borrar ni gestionar configuración. Acceso solo a vistas de Capataz y Carta Balance.

📊

ingeniero

Analista de obra. Lee todo, edita configuración, planes, BIM, LPS. NO puede crear usuarios ni borrar registros. Acceso al Dashboard analítico completo.

🛡️

admin

Control total. Crea/desactiva usuarios, cambia roles, lee auditoría, borra registros. Solo 1-2 personas deben tener este rol (gerente proyecto + tú).

Cómo crear el primer admin

Las reglas requieren un admin para crear usuarios, pero ¿cómo se crea el primero? Manualmente desde Firebase Console:

  1. Crea tu usuario normalmente con email/password en la app
  2. Ve a Firebase Console → Firestore → tu proyecto
  3. Crea colección Usuarios (capital U)
  4. Document ID = el UID del usuario que acabas de crear (lo ves en Authentication → Users)
  5. Agrega los campos: email (string), nombre (string), rol (string: "admin"), activo (boolean: true)
  6. Cierra sesión y vuelve a entrar — ya eres admin

4 Login con protección anti-fuerza-bruta

El componente Login.jsx ahora bloquea después de 5 intentos fallidos por 5 minutos. Mensajes de error genéricos no revelan si el email existe.

Antes vs Después

❌ ANTES
Sin protección
  • "Usuario no existe" → atacante sabe qué emails probar
  • "Contraseña incorrecta" → confirma que el email es válido
  • Sin límite de intentos
  • Sin auditoría de intentos fallidos
✅ AHORA
Endurecido
  • Mensaje único: "Credenciales inválidas"
  • Bloqueo tras 5 intentos por 5 min
  • Countdown visible con tiempo restante
  • Cada intento fallido se audita
  • Botón "🔒 BLOQUEADO" deshabilitado durante cooldown

Verificación de cuenta activa

El nuevo AuthContext.jsx verifica el campo activo al cargar el perfil. Si un admin desactiva una cuenta (activo: false):

  • El usuario es deslogueado automáticamente al refrescar
  • Se le muestra mensaje "Tu cuenta ha sido desactivada"
  • Se registra en Auditoria_Seguridad

7 Cómo desplegar este bloque

El despliegue es CRÍTICO. Hazlo en orden y prueba después de cada paso.

  1. Backup primero: Firebase Console → Firestore → menú ⋮ → Export. Si algo falla puedes restaurar.
  2. Crea tu primer admin manualmente en Firebase Console (paso descrito en §3). Si despliegas las reglas sin tener un admin con perfil válido, te quedas afuera.
  3. Verifica que tu usuario actual tenga documento en /Usuarios/{tu_uid} con { rol: "admin", activo: true }.
  4. Copia los archivos del Bloque 12 a sus rutas:
    • firestore.rules → raíz del proyecto
    • storage.rules → raíz del proyecto
    • firebase.json → raíz (reemplaza el existente)
    • .env.example → raíz
    • src/context/AuthContext.jsx (reemplaza)
    • src/views/Login.jsx (reemplaza)
    • src/utils/seguridad.js (NUEVO)
    • src/components/RoleGuard.jsx (NUEVO)
  5. Despliega Firestore rules: firebase deploy --only firestore:rules
  6. Prueba inmediatamente: abre la app en otra pestaña, recarga. Si tu admin está bien configurado, debes seguir entrando. Si no, REVIERTE las rules con la versión vieja desde Firebase Console.
  7. Despliega Storage rules: firebase deploy --only storage
  8. Build y deploy del frontend: npm run build && firebase deploy --only hosting
  9. Prueba completa: sube una foto, crea un registro, intenta borrar (debe fallar para capataz), sale y entra de nuevo.
  10. Migra usuarios existentes creando documentos en /Usuarios para cada uno (ver script abajo).

Script de migración (Node.js)

Para crear documentos en /Usuarios para usuarios que ya existían en Firebase Auth pero no tenían perfil:

// scripts/migrar-usuarios.js
// Ejecutar UNA SOLA VEZ con `node scripts/migrar-usuarios.js`
const admin = require('firebase-admin');
admin.initializeApp({
  credential: admin.credential.cert(require('./serviceAccountKey.json'))
});

(async () => {
  const { users } = await admin.auth().listUsers();
  for (const u of users) {
    const ref = admin.firestore().doc(`Usuarios/${u.uid}`);
    const snap = await ref.get();
    if (!snap.exists) {
      await ref.set({
        email: u.email,
        nombre: u.displayName || u.email.split('@')[0],
        rol: 'capataz',        // rol seguro por defecto
        activo: true,
        creadoEn: admin.firestore.FieldValue.serverTimestamp(),
      });
      console.log('Migrado:', u.email);
    }
  }
})();
⚠️ ROLLBACK PLAN: Si las reglas te bloquean, ve a Firebase Console → Firestore → Rules → pestaña "History" → click en una versión anterior → "Restore". Las reglas viejas (abiertas) vuelven inmediatamente.
