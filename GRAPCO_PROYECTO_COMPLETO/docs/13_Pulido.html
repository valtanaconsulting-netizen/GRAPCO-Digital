🐛 Bugs detectados y corregidos

Auditoría del código acumulado de los 13 bloques anteriores. Encontré bugs reales que podrían explotar en producción y los corregí antes de cerrar el proyecto.

BUG #1 · CRÍTICO

firebaseConfig duplicado: firebaseConfig.js sin storage exportado

El archivo en uso solo exportaba db y auth. Pero BimUploader.jsx y otros componentes importaban storage desde el mismo path, esperando que existiera. Habría fallado al subir cualquier foto desde producción.

✓ Unificado: nuevo firebaseConfig.js v3 con db + auth + storage + soporte de variables de entorno (.env). Borrado el firebaseConfig_NEW.js duplicado.
BUG #2 · ALTO

URL de Cloud Functions hardcodeada en 3 archivos

apsClient.js, adminClient.js y SaludSistema.jsx tenían cada uno su propia copia de 'https://us-central1-control-productividad-franklin.cloudfunctions.net'. Si cambias de proyecto Firebase, hay que editar en 3 lugares.

✓ Centralizado en nuevo functionsClient.js con FUNCTIONS_BASE_URL configurable por env vars. Los 3 archivos ahora importan de ahí.
BUG #3 · ALTO

Dos sistemas de notificaciones corriendo en paralelo

El Bloque 11 introdujo NotificationProvider, pero el Bloque 12 dejó la línea setToast({...}) Y notify({...}) en showToast(), generando UI duplicada (Toast viejo + Notification nueva).

✓ showToast() ahora solo dispara notify(). Borrado import de Toast viejo y su render. UI consistente.
BUG #4 · MEDIO

Comandos de Cmd+K invisibles para admin

La condición if (!user || rol !== 'ingeniero') return []; ocultaba comandos a los admins, aunque el menú UI sí los incluía.

✓ Cambiado a rol !== 'ingeniero' && rol !== 'admin'.
BUG #5 · MEDIO

ErrorBoundary no logueaba errores a Firebase

Solo hacía console.error. Si un usuario en producción tenía un crash, nadie se enteraba.

✓ ErrorBoundary v2: hace lazy import de Firestore y registra error_no_capturado en Auditoria_Seguridad con stack trace truncado a 1500 chars. Best-effort (try/catch envuelve todo).
BUG #6 · MEDIO

Acción error_no_capturado no estaba en whitelist

El nuevo ErrorBoundary intentaría escribir esa acción pero seguridad.js rechazaba con "Acción no auditable".

✓ Agregada a ACCIONES_AUDITABLES.
BUG #7 · MEDIO

Sin índices Firestore compuestos

Queries con orderBy + where requieren índice compuesto declarado. Sin esto, la query falla con "The query requires an index". Pasa al filtrar Auditoría por UID + ordenar por timestamp, por ejemplo.

✓ Nuevo firestore.indexes.json con 9 índices declarados. Se despliegan con firebase deploy --only firestore:indexes.
BUG #8 · BAJO

.gitignore no protegía credenciales

El gitignore no incluía service account keys ni archivos .env. Riesgo de subir credenciales a git público.

✓ .gitignore reforzado con sección "SECRETOS NUNCA SUBIR" + patrones de Firebase Admin SDK + emulator data + coverage.
BUG #9 · BAJO

UX del ErrorBoundary mejorable

No había forma de reportar el error fácilmente; usuario tenía que tomar screenshot.

✓ Botón "📋 Copiar error" que copia al portapapeles un reporte estructurado: fecha, URL, mensaje, stack, component stack. Listo para pegar en Slack/email.
BUG #10 · BAJO

Detalle técnico del error sin altura máxima

Si el stack trace era muy largo, el modal del ErrorBoundary se hacía gigante.

✓ max-height: 160px + overflow-y: auto en el detalle.
BUG #11 · BAJO

Variables de entorno no documentadas

El Bloque 12 introdujo env.example pero el frontend seguía hardcodeando valores.

✓ firebaseConfig.js ahora lee de import.meta.env.VITE_* con fallback a defaults. functionsClient.js también respeta las env vars.

📐 Refactor: functionsClient.js unificado

Tres archivos tenían el mismo código duplicado para llamar Cloud Functions con auth. Lo extraje a un helper único con manejo de errores estandarizado y timeout configurable.

Antes (3 copias del mismo código)

// apsClient.js
const callFunction = async (nombre, payload, method = 'POST') => {
  const user = auth.currentUser;
  if (!user) throw new Error('Debes iniciar sesión...');
  const idToken = await user.getIdToken();
  // ... 25 líneas más
};

// adminClient.js (otra copia)
const callAdminFn = async (nombre, payload, method = 'POST') => {
  // ... casi idéntica
};

Después (un solo helper compartido)

// functionsClient.js
export async function callFunction(nombre, payload, method, opts) {
  // timeout configurable, manejo de errores estandarizado,
  // auth opcional via flag requireAuth
}

// apsClient.js
import { callFunction } from './functionsClient';
export const obtenerTokenVisor = () => callFunction('apsTokenViewer');

// adminClient.js
import { callFunction } from './functionsClient';
export const crearUsuarioAdmin = (...) => callFunction('adminCrearUsuario', ...);
📊 Impacto: -40 líneas de código duplicado, +1 nuevo archivo de 60 líneas centralizado. Ganancia neta de mantenibilidad. Si mañana cambias el formato de auth, lo cambias en 1 lugar y todos los clientes funcionan.

📖 Documentación final del producto

Dos documentos clave para que el proyecto sea recibible por cualquier persona o equipo:

README.md profesional

Documento estándar de proyecto open-source con: descripción, características, quick start, estructura de carpetas, despliegue, stack técnico, roles, badges. Listo para GitHub.

Manual de Usuario operativo

HTML autocontenido con 3 secciones — una por rol (capataz, ingeniero, admin) — con:

  • Selector visual de rol al inicio
  • Pasos numerados con cards diseñadas
  • Casos de uso comunes ("tu día típico")
  • Atajos útiles y de teclado
  • FAQ por rol con preguntas frecuentes reales
  • Callouts de tips, warnings e info

El manual está pensado para que un nuevo capataz pueda usarlo como onboarding sin necesidad de capacitación presencial. Y un nuevo admin tiene la guía de "cómo crear el primer usuario" + "cómo investigar una operación sospechosa".
