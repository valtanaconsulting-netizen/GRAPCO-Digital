1 ♿ Accesibilidad WCAG AA

Mejoras de accesibilidad que hacen GRAPCO usable por personas con discapacidad visual, motora o cognitiva.

Lo que implementé

  • Skip link — primer elemento del DOM permite saltar al contenido principal con Tab. Esencial para usuarios de lector de pantalla.
  • Focus visible dorado (3px) — todos los elementos focusables muestran un anillo dorado al recibir tab. Visible siempre, no solo en hover.
  • Tamaño táctil mínimo 44px — todos los botones cumplen Apple HIG y Material Design. Reduces errores de tap en móvil.
  • prefers-reduced-motion — si el usuario configuró su SO para reducir animaciones, GRAPCO las desactiva todas automáticamente.
  • forced-colors (Windows High Contrast Mode) — la app respeta el tema de alto contraste de Windows.
  • aria-live, aria-label, role — notificaciones, modales, tooltips y palette tienen los atributos ARIA correctos.
  • Clase .sr-only — utility para texto solo para lectores de pantalla.
  • Tooltips accesibles — soporte focus además de hover (no solo mouse), role="tooltip".

Snippet del CSS de accesibilidad

/* Respeta preferencia de usuarios con sensibilidad al movimiento */
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after {
    animation-duration: 0.01ms !important;
    transition-duration: 0.01ms !important;
  }
}

/* Skip link para navegación con teclado */
.skip-link {
  position: absolute; top: -40px; left: 0;
  background: #1e3a5f; color: #f59e0b;
  padding: 12px 20px; font-weight: 800;
  z-index: 9999;
}
.skip-link:focus { top: 0; }

/* Tamaño mínimo táctil */
button, a[role="button"] { min-height: 44px; }
🎯 Cumplimiento: Estas mejoras llevan GRAPCO de WCAG A básico a WCAG AA, que es el estándar requerido por la mayoría de empresas y gobiernos. Próximo nivel (AAA) requeriría auditorías más profundas de contraste de color en todas las combinaciones.

2 🌙 Dark mode completo

Sistema de temas con 3 modos. Persiste en localStorage y respeta preferencia del SO.

Cómo funciona

❌ ANTES
Solo modo claro

Capataces que trabajan de noche o en obras con luz fuerte tenían que aguantar el blanco que cansa la vista.

✅ AHORA
3 modos: Claro / Oscuro / Sistema

Toggle en el navbar (☀️🌙🖥️). El modo "Sistema" sigue automáticamente la preferencia del SO. Persiste entre sesiones.

Arquitectura

Dos archivos clave:

  • src/context/ThemeContext.jsx (119 L) — Provider con paletas TEMAS.light y TEMAS.dark
  • src/components/ThemeToggle.jsx (95 L) — Botón con dropdown 3 opciones
// Uso desde cualquier componente
import { useTheme, useThemedBase } from '../context/ThemeContext';

function MiComponente() {
  const { temaActivo, esDark, cambiarModo } = useTheme();
  const base = useThemedBase();  // BASE adaptado al tema actual
  return <div style={{ background: base.bg, color: base.text }}>...</div>;
}
💡 Migración progresiva: El hook useThemedBase() permite migrar componentes gradualmente. Los que usan BASE directo siguen funcionando, los nuevos pueden usar la versión themed para soporte automático de dark mode.

3 📱 PWA installable

Los capataces ahora pueden instalar GRAPCO como app nativa en su celular. Funciona offline y se carga más rápido.

Mejoras concretas

  • Service Worker v2 con estrategias diferenciadas: app shell Network First, assets Cache First, APIs Firebase/APS Never Cache (datos siempre frescos)
  • Offline fallback — si no hay conexión, imágenes muestran SVG placeholder en lugar de roto
  • Cleanup automático — al desplegar nueva versión, el SW borra cachés viejas
  • Banner contextual de instalación que aparece en navegadores compatibles
  • Soporte iOS Safari — instrucciones manuales (iOS no soporta beforeinstallprompt)
  • Persistencia inteligente — si usuario rechaza, no volver a mostrar por 7 días
  • Detección "ya instalada" — display-mode standalone

Estrategias de caché del Service Worker

Tipo de recursoEstrategiaPor qué
HTML (app shell)Network FirstGarantiza que siempre tengas la versión más nueva, con fallback offline
JS/CSS/IMG/FONTCache FirstVelocidad: si está cacheado, lo sirve sin tocar la red
Firebase/FirestoreNetwork OnlyDatos sensibles siempre frescos, nunca cacheados
APS Cloud FunctionsNetwork OnlyLo mismo, datos en tiempo real
Imágenes (offline)SVG fallback"Sin conexión" en lugar de rota

4 🔔 Sistema de notificaciones rico

Reemplazo del Toast plano por un sistema con 4 tipos, acciones inline y stack apilable.

❌ ANTES
Toast plano

Solo mostraba un mensaje. Sin tipos visuales, sin acciones, se acumulaban y desaparecían sin historial.

✅ AHORA
NotificationProvider

4 tipos visuales (success/error/warning/info). Acción inline opcional ("REINTENTAR"). Stack máx 5. Historial 50.

Cómo se ve en código

const { notify } = useNotifications();

// Notificación simple
notify({ tipo: 'success', titulo: 'Guardado' });

// Con mensaje y duración
notify({
  tipo: 'error',
  titulo: 'Falló la sincronización',
  mensaje: 'Verifica tu conexión',
  duracion: 6000,
});

// Con acción inline
notify({
  tipo: 'warning',
  titulo: '3 actividades sin reportar',
  accion: { label: 'VER LISTA', onClick: () => abrirPanel() }
});

7 🎯 Storybook visual de componentes

Catálogo HTML standalone que documenta visualmente todos los componentes del Design System.

Por qué un HTML standalone (no Storybook real)

Tomé la decisión de hacer un archivo HTML standalone en vez de instalar Storybook (npm package) por 3 razones:

  1. Cero dependencias nuevas — no agrega ~200 MB de node_modules
  2. Tu sustentación — puedes mostrarlo abriendo el archivo, sin levantar servidor
  3. Capítulo del proyecto — encaja con tu lógica de "cada bloque genera HTML"

🚀 Cómo desplegar este bloque

Pasos concretos para integrar todo en tu proyecto.

Archivos a crear (carpeta context/)

src/context/ThemeContext.jsx          // Provider de tema
src/context/NotificationContext.jsx   // Provider de notificaciones

Archivos a crear (carpeta components/)

src/components/ThemeToggle.jsx        // Botón cambio tema
src/components/PwaInstallPrompt.jsx   // Banner instalar PWA
src/components/Tooltip.jsx            // Tooltip accesible
src/components/CommandPalette.jsx     // Cmd+K palette

Archivos a reemplazar

src/App.jsx              // Wrappeado con providers + Skip link + CommandPalette
src/components/Navbar.jsx // + ThemeToggle integrado
src/views/Ingeniero.jsx   // CPI/EFICIENCIA con Tooltip
src/index.css             // + reglas accesibilidad + dark mode base
public/sw.js              // Service Worker v2

Pasos en orden

  1. Backup primero: haz commit de tu estado actual a Git antes de tocar nada
  2. Crea la carpeta src/context/ si no existe
  3. Copia los 6 archivos nuevos a sus carpetas correspondientes
  4. Reemplaza los 5 archivos modificados con los del bloque
  5. Si tu Vite o build tool requiere registrar el SW de otra forma, ajustar el bloque useEffect en App.jsx
  6. Levanta el dev server con npm run dev
  7. Prueba: presiona Cmd+K → debe abrir la paleta
  8. Prueba: hover en "CPI GLOBAL" → debe aparecer el tooltip explicativo
  9. Prueba: click en el botón de tema en navbar → debe haber 3 opciones
  10. Build production con npm run build → el SW solo se registra en producción
⚠️ Pendiente desde Bloque 9: Las reglas de Firebase siguen allow read, write: if true. Esto es un riesgo de seguridad antes de tu sustentación. Bloque 12 = solo seguridad Firebase.
