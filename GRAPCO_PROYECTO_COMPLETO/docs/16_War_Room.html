🎯 Por qué este bloque importa para la tesis

Hasta el Bloque 16 cada herramienta vivía en su propio silo: el LPS planificaba, la Carta Balance medía, el Optimizador recomendaba. Faltaba el ciclo cerrado.

🎓 Argumento académico: uno de los problemas crónicos de la digitalización en construcción es la fragmentación — cada solución usa su propia base de datos y no comparte con las demás. Esta tesis lo evita gracias a Firestore como single source of truth, y este bloque demuestra cómo se aprovecha esa unicidad para integrar metodologías que en otros proyectos viven separadas.

Lógica de alertas automáticas

// Para cada cuadrilla con >= 10 observaciones:
if (luf < 40) → alerta CRÍTICA
else if (luf < 50) → alerta ALTA
if (tnc > 35) → alerta MEDIA (TNC alto)

// Cada alerta incluye:
- titulo: "Cuadrilla X en CRÍTICO"
- mensaje: "LUF Y% (Z cartas analizadas)"
- accion: "Causa principal: ..." (extraída del Pareto)

Flujo del ingeniero residente

  1. Llega a obra

    Abre la app y va directo al War Room.

  2. Mira KPIs globales

    "5 cuadrillas, LUF agregado 58%, 2 alertas". Sabe el estado en 5 segundos.

  3. Atiende las alertas

    Click en cada alerta: "Cuadrilla Acabados con LUF 35%, causa: espera de instrucciones 100%".

  4. Drill-down

    Abre el modal de la cuadrilla problemática. Ve quiénes son las personas, su rendimiento individual.

  5. Toma acción

    "Hay que coordinar con planificación que las instrucciones lleguen 1 hora antes". Implementa el cambio.

🔁 El ciclo metodológico cerrado

Esta es la verdadera contribución académica: integrar 4 metodologías que normalmente viven aisladas en una sola plataforma con flujo bidireccional de datos.

🗓️ LUNES — Planificación (LPS)

El ingeniero arma el Plan Diario en la pestaña LPS. Para cada actividad, click en "🤖 Sugerir cuadrilla". El sistema mira los datos históricos de Cartas Balance y devuelve la composición óptima.

📐 LUNES TARDE — Replanteo (BIM)

El ingeniero usa el visor BIM para mostrar al capataz qué elemento del modelo se va a construir. Click en el elemento → ver actividad asociada (vínculo WBS↔BIM).

🌅 MARTES — Ejecución (Capataz)

El capataz arma su cuadrilla siguiendo la sugerencia. Reporta avance al final del día con fotos. Datos van a Firestore.

⚖️ MIÉRCOLES — Medición (Carta Balance)

El capataz hace una Carta Balance de 30 minutos. Marca TP/TC/TNC con subcategorías estándar. Datos van a Firestore.

📊 JUEVES MAÑANA — Análisis (War Room)

El ingeniero abre el War Room. Ve el LUF de la cuadrilla, identifica si la sugerencia funcionó, qué causa de TNC apareció.

🎯 JUEVES TARDE — Acción correctiva

Si el LUF fue bajo, el ingeniero coordina con el equipo correcto (logística si fue espera de material, calidad si fue retrabajo, etc.).

📈 VIERNES — Cálculo PPC + EVM (LPS + EVM)

Calcula PPC semanal, registra RNC. Calcula CPI/SPI. Compara con metas. Documenta lecciones aprendidas en LPS.

🔄 SIGUIENTE LUNES — Mejora iterativa

Vuelve al primer paso. Pero ahora el optimizador tiene MÁS datos, hace MEJORES sugerencias, y la cuadrilla está más afilada.

📈 Esto es Mejora Continua (Kaizen) digitalizada. Cada semana el sistema "aprende" más sobre qué cuadrillas funcionan en qué actividades. La sugerencia mejora. El residente toma decisiones cada vez más informadas.
